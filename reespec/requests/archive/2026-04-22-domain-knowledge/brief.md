# Brief: Domain Knowledge

## What

Add a local, persistent domain knowledge system to the reeboot agent — a corpus of source documents (pre-packaged with agent profiles and/or added by the owner) that is vectorised, full-text indexed, and optionally synthesised into an LLM-maintained wiki. The agent queries this corpus at runtime to answer domain-specific questions with citations back to immutable raw sources.

## Why

Today the agent has no persistent knowledge beyond what fits in the context window and what skills/MCP servers provide at runtime. For domain-specific deployments — legal research, medical assistance, client engagement, engineering reference, academic research — the agent needs a body of knowledge that:

- Is always available offline and fast (no external API dependency)
- Grows over time as the owner adds documents
- Builds cross-document connections and synthesis, not just raw retrieval
- Cites authoritative sources, especially in high-stakes domains
- Learns from usage patterns — filed query insights compound the corpus

## Goals

- Owner can drop documents into a `raw/` directory and the agent detects, processes, and indexes them automatically
- Agent detects new documents via file watcher, notifies owner, and offers interactive or silent ingest
- Raw sources are immutable ground truth — the agent reads them but never modifies them
- Documents are chunked, embedded (locally, no API), and stored in the existing SQLite database via sqlite-vec
- Full-text search (FTS5) available alongside vector search for hybrid retrieval
- Each document carries two metadata fields: `source_tier` (provenance, rule-based) and `confidence` (content quality, LLM-assigned)
- Answers always cite raw sources with tier and confidence visible to the owner
- Optional wiki synthesis layer: LLM maintains interlinked markdown pages that sit between the owner and raw sources, compiling cross-document connections, concept pages, and contradiction flags
- Wiki grows from both ingest (source summaries, concept updates) and usage (filed query insights)
- Lint operation (on-demand + scheduled) health-checks the wiki for contradictions, orphans, stale claims, and missing concepts
- Wiki is optional — disabled by default gives pure RAG; enabled adds the synthesis layer on top

## Non-goals

- Live external data sources (legislation APIs, client repos, real-time feeds) — these are skill/MCP concerns, not corpus concerns
- DOCX, DOC, XSLX, PPTX, HTML formats — deferred to v2
- Multi-tenant or per-user corpus isolation — single-owner deployment only
- Cloud-based or API-based embeddings — local ONNX only in v1
- Per-context knowledge bases — corpus is instance-level, shared across all contexts
- Semantic search on wiki pages via embeddings — FTS5 + index.md navigation sufficient in v1

## Two-tier corpus

| Tier | Origin | Location | Mutability |
|---|---|---|---|
| `template` | Ships with agent profile | `raw/template/` | Read-only for owner |
| `owner` | Added by owner at any time | `raw/owner/` | Read-write |

Both tiers feed the same vector index. Provenance is always tracked. A third pseudo-tier `wiki-synthesis` tags pages generated by the LLM from cross-document analysis — always lowest confidence, always points back to raw sources.

## Document metadata

Every ingested document and every wiki page carries two fields:

| Field | Values | Who sets it | What it means |
|---|---|---|---|
| `source_tier` | `template` \| `owner` \| `wiki-synthesis` | Rule-based, automatic | Epistemic distance from raw source |
| `confidence` | `high` \| `medium` \| `low` | LLM-assigned at ingest | Content quality judgement |

These are separate concerns. A template document can be rated low confidence by the LLM (outdated, contradicted). An owner document can be rated high confidence (primary source, well-structured). Wiki synthesis pages are always `wiki-synthesis` tier regardless of content quality.

## Supported formats (v1)

- `.md`, `.txt`, and any plain-text format — read directly
- `.csv` — read and preprocessed (column headers + row context before chunking)
- `.pdf` — text extracted via `pdf-parse`

## Embedding model

`nomic-ai/nomic-embed-text-v1.5` via `@huggingface/transformers` (local ONNX, no server, no API key).

Key properties relevant to this use case:
- 8192 token context window — handles long legal/medical/technical documents
- Matryoshka dimensions — 768 default, configurable down to 256 for storage efficiency
- Task instruction prefixes: `search_document:` for corpus chunks, `search_query:` for user queries — significantly improves RAG retrieval quality
- Downloaded once on first use, cached locally in `~/.cache/huggingface/`

## Ingest flow

**Trigger:** `fs.watch` on `raw/` directory (reuses pi-file-watcher pattern — `fs.watch` + 300ms debounce, close-while-processing, reopen on `agent_end`). New files checked against `knowledge_sources` table by hash — already ingested files are skipped.

**Step 1 — Notify owner:**
```
"I found 2 new document(s) to ingest:
 - contract-acme-2026.pdf (owner)
 - civil-code-amendments.md (template)

Would you like to go through them together, or shall I
process them and send you a summary when done?"
```

**Step 2A — Interactive (default):** One document at a time. Agent reads source, shares key takeaways, flags contradictions with existing corpus, owner steers emphasis. Agent confirms before wiki is updated.

**Step 2B — Silent + notify (owner opt-out):** Agent processes in background. Structured summary delivered via owner's channel when done: pages created/updated, contradictions found, low-confidence items flagged, suggested follow-up questions.

Both paths execute the same underlying ingest pipeline:
1. Extract text (format-dependent)
2. Chunk with overlap
3. Embed chunks with `search_document:` prefix
4. Store in `vec0` virtual table + FTS5 table
5. Assign `source_tier` (rule) + `confidence` (LLM)
6. If wiki enabled: create `wiki/sources/` summary page, update affected concept pages (ripple effect), flag contradictions, update `index.md` + `log.md`

## Retrieval

At query time:
1. Embed query with `search_query:` prefix
2. Vector search → top-k relevant chunks from `vec0`
3. If wiki enabled: read `wiki/index.md` → drill into relevant concept pages
4. Synthesise answer with citations (source name, tier, confidence)
5. Wiki-synthesis citations always flagged: "verify against primary sources"

## Wiki synthesis layer (optional)

When `knowledge.wiki.enabled: true`, the agent maintains a directory of interlinked markdown files:

```
~/.reeboot/knowledge/wiki/
  index.md        — master catalog (updated on every ingest)
  log.md          — append-only activity record
  concepts/       — domain concept pages
  sources/        — per-document summaries
  comparisons/    — filed query insights
```

**Query filing:** After generating a valuable analysis or comparison, the agent offers to file it as a wiki page. Filed pages compound the knowledge base from usage, not just from sources.

**Lint:** Periodic health-check of the wiki. Finds contradictions between pages, orphan pages, concepts mentioned but lacking own page, stale claims superseded by newer sources, low-confidence clusters, and suggests next investigations.

- On-demand: owner asks, or agent proactively suggests ("wiki hasn't been linted in 2 weeks")
- Scheduled: configurable cron, default weekly
- Output: structured report delivered via owner's channel

## Storage

**Filesystem:**
```
~/.reeboot/knowledge/
  raw/template/     — immutable template documents
  raw/owner/        — owner-added documents
  wiki/             — LLM-maintained synthesis (if enabled)
```

**SQLite additions to existing reeboot.db:**
```
knowledge_sources   — path, hash, source_tier, confidence,
                      ingested_at, chunk_count, status
vec0 virtual table  — embeddings (sqlite-vec extension)
FTS5 virtual table  — full-text search on raw content
wiki_pages          — metadata only: path, source_tier,
                      confidence, updated_at, sources[]
                      (content lives in files, not db)
```

## New dependencies

| Package | Purpose |
|---|---|
| `sqlite-vec` | Vector search SQLite extension (pre-v1, Mozilla-backed) |
| `@huggingface/transformers` | Local ONNX embedding inference (no server, no API) |
| `pdf-parse` | PDF text extraction |

Note: `@huggingface/transformers` is also used by Loop 1 (personal memory) if session search is extended to semantic search in a future iteration. Installing it here makes it available to both.

## Config

```json
{
  "knowledge": {
    "enabled": true,
    "embeddingModel": "nomic-ai/nomic-embed-text-v1.5",
    "dimensions": 768,
    "chunkSize": 512,
    "chunkOverlap": 64,
    "wiki": {
      "enabled": false,
      "lint": {
        "schedule": "0 9 * * 1"
      }
    }
  }
}
```

## Impact

- Enables the legal researcher, client engagement, support agent, and academic use cases described in discovery
- Works for any domain — generic by design, domain-specific by corpus
- Wiki disabled by default: zero friction for simple deployments, opt-in sophistication for complex ones
- Builds on Loop 1 (personal memory) infrastructure patterns — same extension model, same scheduler, same SQLite db
- Connects forward to agent profiles request — profiles will declare required knowledge bases as part of their configuration
- Raw sources remain immutable ground truth — hallucination blast radius is bounded and auditable
